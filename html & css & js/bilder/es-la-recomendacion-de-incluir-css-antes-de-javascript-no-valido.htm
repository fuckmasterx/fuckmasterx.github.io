Estado actual de la pregunta

En un sinnúmero de lugares en línea que he visto la recomendación de incluir CSS antes de JavaScript. El razonamiento es, generalmente, de esta forma:

Cuando se trata de ordenar tu CSS y JavaScript, quieres que tu CSS para llegar primero. La razón es que la representación del hilo tiene toda la estilo de la información que necesita para procesar la página. Si el JavaScript incluye, en primer lugar el motor de JavaScript tiene que analizar todo antes de de continuar con la siguiente conjunto de recursos. Esto significa que la representación de el hilo no se puede mostrar por completo la página, ya que no todos los los estilos que necesita.

Mi prueba real revela algo muy diferente:

Mi instrumento de prueba

Yo uso el siguiente script en Ruby para generar retrasos de varios recursos: 

require 'rubygems'
require 'eventmachine'
require 'evma_httpserver'
require 'date'

class Handler  < EventMachine::Connection
  include EventMachine::HttpServer

  def process_http_request
    resp = EventMachine::DelegatedHttpResponse.new( self )

    return unless @http_query_string

    path = @http_path_info
    array = @http_query_string.split("&").map{|s| s.split("=")}.flatten
    parsed = Hash[*array]

    delay = parsed["delay"].to_i / 1000.0
    jsdelay = parsed["jsdelay"].to_i

    delay = 5 if (delay > 5)
    jsdelay = 5000 if (jsdelay > 5000)

    delay = 0 if (delay < 0) 
    jsdelay = 0 if (jsdelay < 0)

    # Block which fulfills the request
    operation = proc do
      sleep delay 

      if path.match(/.js$/)
        resp.status = 200
        resp.headers["Content-Type"] = "text/javascript"
        resp.content = "(function(){
            var start = new Date();
            while(new Date() - start < #{jsdelay}){}
          })();"
      end
      if path.match(/.css$/)
        resp.status = 200
        resp.headers["Content-Type"] = "text/css"
        resp.content = "body {font-size: 50px;}"
      end
    end

    # Callback block to execute once the request is fulfilled
    callback = proc do |res|
        resp.send_response
    end

    # Let the thread pool (20 Ruby threads) handle request
    EM.defer(operation, callback)
  end
end

EventMachine::run {
  EventMachine::start_server("0.0.0.0", 8081, Handler)
  puts "Listening..."
}

Puedo obtener resultados similares en Chrome, Firefox e Internet Explorer. En la Ópera sin embargo, el orden simplemente no importa.

Lo que parece estar sucediendo es que el intérprete de JavaScript se niega a empezar hasta que todos los CSS de su descarga. Por lo tanto, parece que el tener JavaScript incluye a la primera es más eficiente como el JavaScript hilo de hace más tiempo de ejecución.

Me estoy perdiendo algo, es la recomendación de colocar CSS incluye antes de JavaScript incluye no correcto?

Está claro que podríamos añadir async o uso setTimeout para liberar el render de la rosca o de poner el código JavaScript en el pie de página, o uso de JavaScript cargador. El punto aquí es sobre el pedido de esencial JavaScript bits y CSS bits en la cabeza.

  • Aseguró que no había almacenamiento en caché de HTTP, de modo que el navegador tiene que hacer una descarga completa cada vez que se carga una página.
  • Para simular la realidad, yo incluido jQuery y el H5BP CSS (así que hay una cantidad decente de secuencia de comandos/CSS para analizar)

  • Registrado cuánto tiempo tomó para que la secuencia de comandos en línea en la <body> ejecución, que es análoga a DOMReady.

Conclusiones

En primer lugar, es importante tener en cuenta que estoy operando bajo el supuesto de que usted tiene los scripts localizados en el <head> de su documento (en oposición a la final de la <body>). Hay varios argumentos acerca de por qué podría vincular a los scripts de la <head> versus el final del documento, pero que está fuera del alcance de esta respuesta. Esto es estrictamente acerca de si <script>s debe ir antes de la <link>s en <head>.

En los navegadores actuales, se ve como la vinculación a CSS primera nunca proporciona un aumento del rendimiento. Poner CSS después de la secuencia de comandos que recibe una mínima cantidad de la ganancia cuando ambos script y CSS se retrasa, pero da grandes ganancias al CSS se retrasa. (Se muestra el last columnas en el primer conjunto de resultados).

Dado que la vinculación de CSS último no parece perjudicar el rendimiento, pero puede proporcionar ganancias bajo ciertas circunstancias, usted debe enlace externo hojas de estilo , después de que link para scripts externos sólo en los navegadores de escritorio, si el rendimiento de los navegadores antiguos no es una preocupación. Leer en para el móvil de la situación.

¿Por qué?

Históricamente, cuando un navegador se encontró con un <script> etiqueta apuntando a un recurso externo, el navegador podría dejar de analizar el HTML, recuperar la secuencia de comandos, ejecutar, a continuación, continuar analizando el código HTML. En contraste, si el navegador que se encontró con un <link> para una hoja de estilos externa, sería continuar con el análisis del HTML mientras que trajo el archivo CSS (en paralelo).

Por lo tanto, el ampliamente repetidos consejos para poner las hojas de estilo de primera que iba a descargar en primer lugar, y el primer script para descargar podría ser cargado en paralelo.

Sin embargo, los navegadores modernos (incluyendo todos los navegadores que he probado anteriormente) han implementado especulativo de análisis, donde el navegador "mira hacia delante" en el HTML y comienza la descarga recursos antes de descargar y ejecutar secuencias de comandos.

En navegadores antiguos, sin especulativo de análisis, poniendo guiones primero va a afectar a su rendimiento, ya que no se descarga en paralelo.

Soporte De Navegadores

Especulativo de análisis fue implementada por primera vez en: (junto con el porcentaje de todo el mundo navegador de escritorio a los usuarios el uso de esta versión, o mayor, como de enero de 2012)

  • Chrome 1 (WebKit 525) (100%)
  • IE 8 (75%)
  • Firefox 3.5 (96%)
  • Safari 4 (99%)
  • Opera 11.60 (85%)

En total, aproximadamente el 85% de los navegadores de escritorio en uso hoy en día el apoyo especulativo de carga. Poner los scripts antes de CSS tendrá una penalización de rendimiento en un 15% de usuarios a nivel mundial; YMMV basada en su sitio audiencia específica. (Y recuerde que el número se está reduciendo.)

En los navegadores de móviles, es un poco más difícil llegar definitiva números, simplemente debido a lo heterogéneo que el navegador móvil y OS paisaje. Desde especulativo de representación se implementó en WebKit 525 (publicado Mar 2008), y casi todos los que vale la pena navegador móvil basado en WebKit, podemos concluir que "la mayoría" de los navegadores para móviles debe de apoyo. Según quirksmode, iOS 2.2/Android 1.0 utilizar WebKit 525. No tengo idea de lo que Windows Phone parece.

Sin embargo, me encontré con la prueba en mi dispositivo Android 4, y mientras yo veía los números similar a la de escritorio resultados, me conecté a la fantástica nueva depurador remoto de Chrome para Android, y la ficha network (Red) mostró que el navegador era en realidad la espera para descargar el CSS hasta la JavaScripts totalmente cargada - en otras palabras, incluso en la versión más reciente de WebKit para Android no parece apoyar especulativo de análisis. Yo sospecho que podrían estar desactivados debido a la CPU, la memoria y/o la red de las limitaciones inherentes a los dispositivos móviles.

Hay dos razones principales para poner CSS antes de JavaScript.

  1. Edad de los navegadores (Internet Explorer 6 y 7, Firefox 2, etc.) iba a bloquear todas las descargas siguientes cuando se inició la descarga de un script. Así que si usted tiene a.js seguido por b.css que se descargan de forma secuencial: primero una luego la b. Si usted tiene b.css seguido por a.js que se descargan en paralelo de modo que la página cargue más rápidamente.

  2. Nada se procesa hasta que todas las hojas de estilo son descargados - esto es cierto en todos los navegadores. Los Scripts son diferentes, ellas bloque de representación de todos los elementos del DOM que están por debajo de la etiqueta de secuencia de comandos en la página. Si usted pone su scripts en la HEAD, entonces eso significa que toda la página está bloqueada desde la representación hasta que todas las hojas de estilo y los scripts que se descargan. Si bien tiene sentido para bloquear todo el procesamiento de las hojas de estilo (para obtener el estilo correcto la primera vez y evitar el flash de contenido sin estilo FOUC), no tiene sentido para bloquear la representación de la totalidad de la página para secuencias de comandos. A menudo, los scripts no se afectará ninguna de elementos del DOM o solo una parte de los elementos del DOM. Lo mejor de la carga de secuencias de comandos como la baja en la página como sea posible, o mejor aún, ellos cargan de forma asincrónica.

Es divertido crear ejemplos con Cuzillion. Por ejemplo, esta página tiene una secuencia de comandos en la HEAD por lo que toda la página está en blanco hasta que se hace la descarga. Sin embargo, si nos movemos de la secuencia de comandos para el final del CUERPO del bloque de la cabecera de la página hace desde los elementos DOM producirse por encima de la etiqueta de secuencia de COMANDOS, como se puede ver en esta página.

No me gustaría enfatizar demasiado en los resultados que tienes, creo que es subjetivo, pero tengo una razón para explicar que es mejor poner en el CSS antes de js.

Durante la carga de tu sitio web, existen dos escenarios que se podrían ver:

CASO 1: pantalla en blanco > sin estilo web > estilo de página web > interacción > estilo de página web interactiva y

CASO 2: pantalla en blanco > sin estilo web > interacción > estilo de página web > estilo de página web interactiva y

Honestamente, no puedo imaginar a nadie que la elección de Caso 2. Esto significaría que los visitantes usando conexiones lentas a internet, tendrá que enfrentar a un sitio web sin estilo, que les permite interactuar con ella utilizando Javascript (puesto que ya está cargado). Además, la cantidad de tiempo que pasan buscando en un sitio web sin estilo se multiplicarían de esta manera. ¿Por qué alguien querría eso?

También funciona mejor como jQuery estados

"Cuando el uso de los guiones que se basan en el valor de estilo CSS propiedades, es importante la referencia de estilo externas o incrustación de estilo los elementos antes de hacer referencia a las secuencias de comandos".

Cuando los archivos se cargan en el orden incorrecto (primera JS, luego CSS), todo el código Javascript depender de las propiedades establecidas en ficheros CSS (por ejemplo, la anchura o la altura de un div) no se haya cargado correctamente. Parece que con el mal orden de carga, las propiedades correctas son 'a veces' conoce a Javascript (tal vez esto es causado por una condición de carrera?). Este efecto parece más grande o más pequeña dependiendo del navegador utilizado.

Fueron las pruebas realizadas en su computadora personal o en un servidor web? Es una página en blanco, o es un complejo sistema en línea con imágenes, bases de datos, etc.? Son los scripts de realizar un simple evento hover acción, o son un componente básico de cómo su sitio web hace que interactúa con el usuario? Hay varias cosas a considerar aquí, y la relevancia de estas recomendaciones casi siempre se convierten en reglas cuando se aventura en el alto calibre de desarrollo web.

La finalidad de las "hojas de estilo en la parte superior y las secuencias de comandos en la parte inferior de la regla" es que, en general, es la mejor manera de lograr un óptimo procesamiento progresivo, que es crucial para la experiencia del usuario.

Más allá de todo: asumiendo que la prueba es válida, y que realmente se están produciendo resultados contrarios a las populares reglas, que iba a venir como ninguna sorpresa, la verdad. Cada sitio web (y de todo lo que se necesita para hacer que todo aparezca en la pantalla del usuario) es diferente y el Internet está en constante evolución.
